## 1. Continuous Auditing Engine

**Role:** Lead Data Engineer & Developer  
**Duration:** 2+ years  
**Industry:** Pension Fund Administration (Financial Services)

### Problem Statement

The internal audit team needed to transition from periodic manual reviews to a **continuous monitoring system** capable of executing hundreds of business rules daily against the core transactional database, generating automated alerts and dashboards.

### Architecture

```
┌─────────────────┐     ┌──────────────────┐     ┌─────────────────┐
│  Oracle DB      │────▶│  ETL Engine      │────▶│  SQL Server     │
│  (Source)       │     │  (Pentaho PDI)   │     │  (Data Mart)    │
└─────────────────┘     └──────────────────┘     └────────┬────────┘
                                                          │
                        ┌──────────────────┐              │
                        │  Rules Engine    │◀─────────────┘
                        │  (T-SQL / PL/SQL)│
                        └────────┬─────────┘
                                 │
                   ┌─────────────┼─────────────┐
                   ▼             ▼             ▼
            ┌───────────┐ ┌───────────┐ ┌───────────┐
            │  Email    │ │  Power BI │ │  Excel    │
            │  Alerts   │ │  Dashboard│ │  Reports  │
            └───────────┘ └───────────┘ └───────────┘
```

### Technical Implementation

- **Data Extraction:** Designed ETL pipelines using Pentaho PDI to extract data from Oracle (core banking system) into a SQL Server data mart with a star schema design
- **Rules Engine:** Built 100+ validation rules in T-SQL and PL/SQL covering:
  - Age and gender validation
  - Balance reconciliation
  - Transaction anomaly detection
  - Regulatory compliance checks
  - Cross-system data consistency
- **Notification System:** Automated email alerts with HTML-formatted reports generated by Pentaho, triggered by rule violations
- **Dashboard:** Power BI dashboards showing rule execution status, violation trends, and resolution tracking
- **Data Model:** Designed a dimensional model with fact tables (`fact_transactions`, `fact_rule_results`) and dimension tables (`dim_rules`, `dim_dates`, `dim_categories`)

### Technologies

| Category | Technologies |
|----------|-------------|
| ETL | Pentaho PDI (Jobs + Transformations) |
| Databases | Oracle 19c, SQL Server 2019 |
| Languages | PL/SQL, T-SQL, Python, JavaScript |
| BI | Power BI |
| Notifications | SMTP, HTML templates |

### Key Metrics

- **100+ business rules** automated
- **Daily execution** across millions of records
- **80% reduction** in manual audit time
- **Real-time visibility** through Power BI dashboards

### Lessons Learned

- Star schema design significantly improved query performance for rule execution
- Parameterized rules (thresholds in dimension tables) enabled business users to adjust without code changes
- Pentaho PDI's job orchestration with error handling and retry logic proved essential for production reliability

## 2. Pension Reform Data Integration

**Role:** Data Engineer  
**Duration:** 1+ year  
**Industry:** Pension Fund Administration

### Problem Statement

A major pension reform required integrating data flows between pension fund administrators (AFP) and the national social security institute (IPS), involving complex file validation, cross-referencing, and regulatory reporting.

### Architecture

```
┌──────────────┐     ┌──────────────────┐     ┌──────────────────┐
│  AFP Systems │────▶│  Validation      │────▶│  IPS Integration │
│  (Oracle)    │     │  Engine          │     │  Files           │
└──────────────┘     │  (SQL + Pentaho) │     └──────────────────┘
                     └────────┬─────────┘
                              │
                     ┌────────▼─────────┐
                     │  Reconciliation  │
                     │  Reports (Excel) │
                     └──────────────────┘
```

### Technical Details

- **Intra-validation:** Cross-field validation within each data file (200+ validation rules)
- **Inter-validation:** Cross-file validation between AFP and IPS datasets
- **Reconciliation:** Balance and record count reconciliation between systems
- **File Format:** Structured flat files with specific field-level validation per regulatory specification
- **Error Handling:** Detailed error logging with categorized rejection codes

- **15+ regulatory file types** processed

- **Automated daily processing** of millions of pension records

## 3. AML Compliance Rules Engine

**Role:** Data Engineer  
**Duration:** 6+ months  
**Industry:** Financial Services (Regulatory Compliance)

### Problem Statement

Develop a rules engine for **Anti-Money Laundering (AML)** detection, implementing casuistic analysis on deposit/withdrawal patterns across voluntary savings accounts to identify suspicious transactions.

### Architecture

```
┌──────────────────┐     ┌──────────────────┐     ┌──────────────────┐
│  Data Warehouse  │────▶│  AML Rules       │────▶│  Alerts &        │
│  (SQL Server)    │     │  Engine (T-SQL)  │     │  Reports         │
└──────────────────┘     └──────────────────┘     └──────────────────┘
```

### Casuistic Analysis Patterns

The rules engine detects patterns including (generalized):

1. **Income vs. Deposit ratio anomalies** — deposits significantly exceeding declared income
2. **High-value foreign currency transactions** — transactions above regulatory thresholds in non-local currency
3. **Underage account activity** — unusual transaction patterns in accounts belonging to minors
4. **Independent worker patterns** — specific patterns associated with self-employed individuals
5. **Rapid deposit-withdrawal cycles** — funds deposited and withdrawn within short timeframes
6. **Multi-account concentration** — patterns spread across multiple account types

### Technical Details

- **Stored Procedures:** Complex T-SQL stored procedures using window functions (`LEAD`, `LAG`, `ROW_NUMBER`), CTEs, and temporal joins
- **Threshold Configuration:** Parameterized thresholds stored in configuration tables for regulatory flexibility
- **Historical Analysis:** Lookback periods configurable by rule type (30, 60, 90, 180 days)

### Technologies

T-SQL, SQL Server, Window Functions, CTEs, Data Warehouse

## 5. Contact Center Analytics Platform

**Role:** Automation Engineer  
**Duration:** 1+ year  
**Industry:** Financial Services

### Problem Statement

Multiple contact center platforms (ticketing, telephony, quality management) required unified data extraction for operational analytics and regulatory reporting.

### Solution

Built a **multi-platform web scraping framework** using Selenium WebDriver with:

- **External configuration** — credentials and parameters in JSON config files, not hardcoded
- **Retry logic** — automatic retries with exponential backoff for transient failures
- **Multi-browser support** — Chrome and Edge WebDriver compatibility
- **Structured output** — JSON and CSV exports with standardized schemas
- **Scheduling integration** — Pentaho PDI jobs triggering Python scripts

### Platforms Integrated

| Platform | Data Extracted |
|----------|---------------|
| Ticketing System | Ticket history, resolution times, SLA compliance |
| Telephony Platform | Call logs, durations, agent performance |
| Quality Management | Evaluation scores, compliance metrics |
| Regulatory Portal | Complaints, regulatory actions, response tracking |

### Technologies

Python, Selenium WebDriver, BeautifulSoup, argparse, JSON config, Pentaho PDI

## 6. Automated Email Processing

**Role:** Automation Developer  
**Duration:** 3+ months  
**Industry:** Financial Services

### Problem Statement

Automate the extraction of attachments from specific email workflows for data governance and compliance processes.

### Solution

- **Outlook COM integration** — Python scripts using `win32com.client` to access local Outlook mailboxes
- **Exchange Online integration** — Alternative implementation using `exchangelib` for server-side access
- **Filtering logic** — Subject-based and sender-based email filtering
- **Attachment handling** — Automatic download with duplicate detection and date-based organization
- **Error handling** — Graceful handling of connection failures and locked mailboxes

### Technologies

Python, win32com (MAPI), exchangelib, Outlook, Exchange Online
